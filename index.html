OC block实现原理

发表于 2017-05-02

block

iOS中经常出现的一个知识点就是block，也就是闭包。block最早出现在iOS4中，并在后续的版本中大量的推广和使用了这项技术,如视图动画API的改版，GCD技术等。block本质上就是闭包功能在iOS上的实现而已。而闭包功能在其他很多语言中都有实现。用闭包可以解决执行逻辑和上下文韩静解耦的场景，如果从设计模式的角度来考虑的话，闭包就是一种策略模式的实现
本文主要是探讨OC的block机制是如何实现的。如下一段含有block的OC代码：

#import <Foundation/Foundation.h>
void test()
{
        int a = 10;
    __block int b = 20;
    NSString *str = @"123";
    __block NSString *blockStr = str;
    NSString *strongStr = @"456";
    __weak NSString *weakStr = @"789";

    void (^testBlock)(int) = ^(int c){
        int d = a + b + c;
        NSLog(@"d=%d, strongStr=%@, blockStr=%@,     weakStr=%@", d, strongStr, blockStr, weakStr);
       };

    a = 20;
    b = 40; 
    testBlock(30);  
}

输出结果如下：

d=80, strongStr=456, blockStr=123, weakStr=789

上面的代码片段中，我们分别定义了：

  • 不带修饰符的基本类型变量a
  • 带__block修饰符的block变量b和blockStr
  • 默认带__strong修饰符的变量strongStr
  • 带__weak修饰符的变量weakStr
    这些修饰符关键字的使用会对block块内的diamante在运行时产生不同的影响。就上面的代码片段而言，当我们在编译时，编译器到底做了什么处理？如果能够了解到编译器的编译过程，那么对我们掌握其实现机制就非常有帮助。我们可以借助命令来看到中间过程，如下：

clang -rewrite-objc -fobjc-arc -stdlib=libc++ -mmacosx-version-min=10.7 -fobjc-runtime=macosx-10.7 -Wno-deprecated-declarations test.m
clang命令会在相同目录下产生一个test.cpp文件。这个文件是OC代码的C++实现版，因为C++是不支持闭包技术的，因此可以通过查看test.app文件来了解OC中的闭包技术到底是如何用函数和结构体来实现的。以下为test.app部分实现：

对上面的代码进行简化和处理，去掉一些次要的东西，就可以简化为如下的代码：

//  每个block变量都会生成一个和OC类内存结构兼容的结构体。下面是_block int b 的结构体定义：
struct Block_b {
    void *isa;    //固定为NULL
      Block_b *forwarding;  //指向真正的block对象变量。
      int flags;
   int size; //结构体的size
      int b;   //保存代码中定义的变量值。
};

//每个block变量都会生成一个和OC类内存结构兼容的结构体。下面是     __block NString *blockStr 的结构体定义：
struct Block_blockStr {
    void *isa;  //固定为NULL
    Block_blockStr *forwarding;  //指向真正的block对象变量
    int flags;
    int size;  //结构体的size
    NSString * blockStr;  //保存代码中定义的变量值。
};


//每个block块都会生成一个和OC类内存结构兼容的结构体和一个描述这个block块信息描述的结构体
struct Block_testBlock {
    //所有block块的固定部分，这也是一个OC类的内存结构。
    Class isa;  //block的OC类型
    int flags;
    int reserved;
    void *funcPtr;       //block块函数的地址。
    Block_testBlock_Desc* desc;   //block的描述信息。

    //所有在block代码块内引用的外部数据都会成为结构体内的数据成员。
    int a;
    NSString * strongStr;
    NSString * __weak weakStr;
    Block_b *b;
    Block_blockStr *blockStr;

    //结构体的构造函数。
    Block_testBlock(void *_funcPtr, Block_testBlock_Desc *_desc, int _a, NSString * _strongStr, NSString * _weakStr, struct Block_b *_b, Block_blockStr *_blockStr, int _flags)
    {
        isa = &_NSConcreteStackBlock;  //根据具体的block类型赋值。
        flags = _flags;
        reserved = 0;
        funcPtr = _funcPtr;
        desc = _desc;
        a = _a;
        strongStr = _strongStr;
        weakStr = _weakStr;
        b = _b->forwarding;  //b保存真实的block变量的地址。
        blockStr = _blockStr->forwarding;  //blockStr保存真实的block变量的地址。
    }
};

    //block块信息描述的结构体定义，主要有block对象的尺寸，以及block中函数的参数信息，也就是参数的签名信息。并生成一个全局的常量对象_testBlock_desc_DATA

轻量包管理工具Carthage

发表于 2017-03-28

我们熟知的管理工具有cocoaPods，cocoaPods 是一整套的 包管理方案，在podfile中配置好我们需要的第三方库，然后cocoapods会进行下载，集成，然后修改我们项目的workspace文件，这一系列操作，

相比之下，carthage非常轻量，它会有一个cartfile描述文件，但是Carthage不会对项目结构进行更改，他的使用只是根据我们的描述文件，对配置的第三方库进行下载，并且使用xcodebuild命令构建成frameWork文件，然后由我们自己将这些库集成到项目中，carthage使用的是一种非侵入性的哲学。

但是这样一来，carthage就没有cocoaPods的集成操作能力了，开发者必须在carthage下载并构建完第三方库后，手动的关联到项目中， 另外，carthage除了非侵入式，他还是去中心化的，它的包管理不像cocoapods那样，有一个中心的服务器来管理各个包的元信息，(cocoapods中的包，如果我们想要上传自己写的库，还需要配置相应的podspec文件，并且用pod命令push上去由cocoapods团队进行审核)。不会受中心服务器信息量和稳定性的限制，弊端的话，就是我们想查找第三方库的时候，也没有一个中心服务器来帮助我们索引，必须自行从网络进行查找。

如何使用：首先，我们需要安装Carthage环境，可以从它的github主页上找到下载地址：https://github.com/Carthage/Carthage/releases
下载pkg文件，安装，终端下进入项目根目录。
